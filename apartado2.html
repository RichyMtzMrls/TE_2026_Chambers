1. ¿Qué es un patrón de diseño?

Un patrón de diseño es una solución general y reutilizable a un problema que se repite frecuentemente en el diseño de software. No es un diseño terminado que pueda transformarse directamente en código; es una descripción o plantilla de cómo resolver un problema que puede usarse en muchas situaciones diferentes.

Los patrones de diseño son las mejores prácticas formalizadas que el programador puede utilizar para resolver problemas comunes cuando diseña una aplicación o sistema. Aceleran el proceso de desarrollo al proporcionar paradigmas probados y testeados.

Cada patrón consta de cuatro elementos esenciales:

  • Nombre: identifica el patrón de forma concisa.
  • Problema: describe cuándo aplicar el patrón.
  • Solución: describe los elementos que forman el diseño.
  • Consecuencias: resultados y compromisos de aplicar el patrón.

2. Origen e historia

El concepto fue introducido originalmente por el arquitecto Christopher Alexander en su obra A Pattern Language (1977), donde documentó patrones para el diseño de edificios y ciudades.

"Un patrón describe un problema que ocurre una y otra vez en nuestro entorno, y luego describe el núcleo de la solución a ese problema."
— Christopher Alexander, 1977

En 1994, Erich Gamma, Richard Helm, Ralph Johnson y John Vlissides —conocidos colectivamente como el Gang of Four (GoF)— publicaron Design Patterns: Elements of Reusable Object-Oriented Software, que cataloga 23 patrones clásicos. Este libro estableció el vocabulario común que los desarrolladores usan hasta hoy.

3. Las tres categorías

Los 23 patrones clásicos del GoF se agrupan en tres familias según su propósito:

Creacionales

Gestionan los mecanismos de creación de objetos, buscando crearlos de manera adecuada a la situación. Abstraen el proceso de instanciación y ayudan a hacer el sistema independiente de cómo se crean sus objetos.

Estructurales

Se ocupan de la composición de clases y objetos para formar estructuras más grandes y flexibles. Describen cómo combinar clases y objetos para obtener nuevas funcionalidades.

De comportamiento

Identifican patrones de comunicación y distribución de responsabilidades entre objetos en un sistema. Se centran en los algoritmos y la asignación de responsabilidades entre objetos.

4. Patrones principales

Creacionales

Singleton
Garantiza que una clase tenga una única instancia y provee un punto de acceso global a ella.
Factory Method
Define una interfaz para crear un objeto, pero deja que las subclases decidan qué clase instanciar.
Abstract Factory
Provee una interfaz para crear familias de objetos relacionados sin especificar sus clases concretas.
Builder
Separa la construcción de un objeto complejo de su representación, permitiendo el mismo proceso crear distintas representaciones.
Prototype
Especifica los tipos de objetos a crear usando una instancia prototípica, y crea nuevos objetos copiando ese prototipo.

Estructurales

Adapter
Permite la colaboración entre objetos con interfaces incompatibles actuando como traductor entre ellos.
Decorator
Añade responsabilidades a un objeto dinámicamente, como alternativa flexible a la herencia para extender funcionalidades.
Facade
Provee una interfaz simplificada a un subsistema complejo de clases, ocultando su complejidad interna.
Composite
Compone objetos en estructuras de árbol para representar jerarquías parte-todo.

De comportamiento

Observer
Define una dependencia uno-a-muchos de modo que cuando un objeto cambia de estado, todos sus dependientes son notificados automáticamente.
Strategy
Define una familia de algoritmos, encapsula cada uno y los hace intercambiables entre sí.
Command
Encapsula una solicitud como un objeto, permitiendo parametrizar clientes con distintas solicitudes, hacer cola o llevar un registro de ellas.
Iterator
Provee una manera de acceder secuencialmente a los elementos de un objeto agregado sin exponer su representación interna.

5. Ejemplo práctico: Singleton

El patrón Singleton asegura que solo exista una instancia de una clase a lo largo de toda la ejecución del programa:


# Implementación del patrón Singleton en Python

class DatabaseConnection:
    _instance = None

    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance.connect()
        return cls._instance

    def connect(self):
        self.connection = "Conexión activa"
        print("Nueva conexión establecida")

# Uso — ambas variables apuntan a la misma instancia
db1 = DatabaseConnection()
db2 = DatabaseConnection()

print(db1 is db2)  # True

Sin importar cuántas veces se llame a DatabaseConnection(), siempre se retorna la misma instancia, evitando conexiones duplicadas innecesarias.

6. Principios SOLID relacionados

Los patrones de diseño están profundamente vinculados a los principios SOLID, base de un software bien estructurado:

S — Single Responsibility
Una clase debe tener una sola razón para cambiar.
O — Open / Closed
Abierto para extensión, cerrado para modificación.
L — Liskov Substitution
Las subclases deben ser sustituibles por sus clases base sin alterar el comportamiento del programa.
I — Interface Segregation
Prefiere interfaces pequeñas y específicas sobre interfaces grandes y genéricas.
D — Dependency Inversion
Depende de abstracciones, no de implementaciones concretas.
